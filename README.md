# HSC Samples

HSC Samples is a small client-side web app for collecting field sample data, organizing it by project, and producing a printable/exportable chain-of-custody style report.

The app is built as a static HTML/CSS/JavaScript site. Most day-to-day data is stored in the browser with `localStorage`, and an optional cloud sync flow uses Back4App via the Parse JavaScript SDK.

## What It Does

- Create and edit projects.
- Add, edit, and delete samples within a project.
- Capture sample metadata such as date, location/person, badge type, serial number, duration, and requested analyses.
- Scan QR codes into the serial number field using the device camera.
- Generate a report view for printing.
- Export the report table to Excel.
- Store company/user metadata for report headers.
- Optionally sync projects to Back4App after login.

## Main Screens

- `index.html`: Home screen and main navigation.
- `project.html`: Project editor and sample entry workflow.
- `project_list.html`: Saved project browser with filtering and delete support.
- `report.html`: Printable/exportable report page.
- `admin.html`: Company and sampler information.
- `login.html`: Parse/Back4App login and cloud sync.

## How Data Is Stored

By default, project data is stored in browser `localStorage`.

- Each project is saved under its project name as the storage key.
- The active project name is stored under `currentProjectName`.
- Company metadata is stored under `companyData`.
- Deleted synced object IDs are tracked under `deleteList` for cloud cleanup.

Because the app is browser-storage based, data is tied to the browser/profile unless you use cloud sync.

## Quick Start

This repo does not require a build step.

1. Serve the repository with a local web server.
2. Open `index.html` through that server in a browser.
3. Create a project and begin entering samples.

Example with Python:

```bash
python -m http.server 8000
```

Then open:

```text
http://localhost:8000/
```

## Why Use a Local Server

Running through `http://localhost` is recommended because:

- camera-based QR scanning is more reliable in a secure context,
- service worker and PWA behavior generally expect HTTP(S),
- opening files directly with `file://` can cause browser restrictions.

## Cloud Sync

Cloud sync is implemented in `scripts/login.js` using Parse with Back4App.

Important notes:

- The Parse application keys are currently embedded in client-side JavaScript.
- Users must log in before the sync button is enabled.
- Sync pushes locally stored projects, deletes remote objects queued in `deleteList`, and then hydrates local storage from the cloud for the current user.

If you plan to use this in production, review the client-side credential exposure and Parse security rules carefully.

## Evaluation

### Strengths

- Very lightweight and easy to host.
- No build pipeline required.
- Clear page separation for the main workflows.
- Works offline-first for core data entry because of local storage.
- Includes practical field features like QR scanning and printable output.

### Current Limitations

- No automated tests or build validation.
- Business logic is embedded in page-level scripts with global variables.
- Data persistence depends on the browser unless cloud sync is used.
- Parse keys are exposed client-side.
- The report/export flow is browser-driven rather than generated by a backend service.
- Some files and strings suggest prototype-stage behavior and inconsistent encoding.

### Good Next Improvements

- Move configuration such as Parse credentials into a safer deployment model.
- Add input validation for project/sample data.
- Separate storage, sync, and UI concerns into reusable modules.
- Add a small test harness for critical behaviors like save/load/report generation.
- Normalize text encoding and clean up legacy/example assets.
- Document the expected deployment target for the service worker and manifest paths.

## Notes

- `requirements.txt` is not a Python dependency lockfile here; it contains product requirements and implementation notes.
- The PWA configuration in `manifest.webmanifest` and `sw.js` appears tailored to hosting under `/hsc_samples`, such as GitHub Pages or a subpath deployment.
